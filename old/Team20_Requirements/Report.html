Analysis Paragraph

For this project, we were given the task of implementing the Chocoholics Anonymous program. Our submitter (Madison Scott) and other team members have curated a report entitled (Report.html) containing vital information regarding the requirements of the project that were gleaned from the description of the project. Our task involves writing the ChocAn data processing software. Other organizations are responsible for the communications software, designing the ChocAn provider’s terminal, for the software needed by Acme Accounting Services, and for implementing the EFT component. For the sake of the project, we have assumed that operators and managers need to be operators and managers need to be validated and operators need to enter a provider number or member number to edit or delete their information in the records. After a careful examination of the project description, we carefully crafted a glossary that defined all of the key terms employed in our use case diagram, such as "Provider Directory". We then designed use case diagrams that displayed all of the use cases we derived from the program and the relationships between them. All of our members created their own, as shown in the picture below. We then refined our original diagrams and created a single use case diagram for our team, which is shown below. We then collaborated to write detailed lists of the steps needed to achieve all of our use cases, including what would occur if errors were made by actors. Finally, we made sure to employ BitBucket in our collaboration and shared all of our files in the team repository, as shown below in our BitBucket statistics.

Use Case: Verify Manager
Context: An manager intends to verify themselves in order to access the system.
Actors: Manager
Main Success Scenario:
  1. The system prompts the manager to confirm their role.
  2. The operator enters their manager password to validate their role.
  3. The system displays a message confirming that the manager has been verified.
  4. The use case ends.
Extensions:
    2a. The manager enters an invalid password.
       2a.1. The system displays a message stating that the password is incorrect.
       2a.2.The use case resumes at Step 1.

Use Case: Print Provider Report extends Generate Reports
Context: A report detailing a provider's identifiable information and a summary of their work for the week is generated.
Actors: Manager
Main Success Scenario:
    5.1. The manager selects "Generate Provider Report".
    5.2. The system reads the week's file of services provided.
    5.3. A system prompts the manager the enter a provider number.
    5.4. The system validates the authentication.
       5.4.1. The system communicates with the ChocAn Data Center to validate the provider number.
    5.5. A report is printed containing: individual provider name (25 characters), provider number (9 digits), provider street address (25 characters), provider city (14 characters), provider state (2 letters), provider ZIP code (5 digits), and for each service provided, date of service (MM–DD–YYYY), date and time data were received by the computer (MM–DD–YYYY HH:MM:SS), member name (25 characters), member number (9 digits), service code (6 digits), fee to be paid (up to $999.99), the total number of consultations with members (3 digits), and the total fee for week (up to $99,999.99).
    5.6. The report is sent to the manager.
    5.7. The system exits.

Use Case: Run Main Accounting Procedure
Context: A number of reports are printed and sent to their respective entities on at midnight on a Friday.
Actors: Timer
Main Success Scenario:
  1. At midnight on Friday, the main accounting procedure is activated by a timer.
  2. The procedure reads the week's file of services provided from the ChocAn Data Center.
  3. A report containing a member's personally identifiable information and a list of the services they were provided is generated for all members who consulted ChocAn for that week.
  4. Each report is sent to its respective member as an email attachment.
  5. A report containing a provider's personally identifiable information, a list of services he or she provided to ChocAn members, and details regarding their compensation is generated for all providers who have billed Choc during that week.
  6. Each report is sent to its respective provider as san email attachment.
  7. A record consisting of electronic transfer funds data is generated and written to a disk.
  8. A report summarizing the accounts payable for the week is generated.
  9. The summary report is sent to the manager as an email attachment.
  10. The accounting procedure ends and the system exits.
Extensions:
  1. N/A

Use Case: Member Validation
Context: A provider validates a member receiving a service.
Actors: Provider
Main Success Scenario:
  1. The provider takes the member’s card and slides it though the terminal’s card reader.
  2. The card number is verified.
    3. 2.1. The terminal dials the ChocAn Data Center.
    2.2. The data center’s computer verifies the member number.
    2.3. The word “Validated” appears on the terminal.
  3. The system exits.
Extensions:
    2a. The card number is invalid.
       2a.1. The ChocAn Data Center could not verify the card number.
       2a.2. “Invalid number” displays on the terminal.
       2a.3. The use case resumes at Step 3.
    2b. The member is suspended.
       2b.1. The ChocAn Data Center flagged the member number as having a suspended account.
       2b.2. “Member Suspended” displays on the terminal.
       2b.3. The use case resumes at Step 3.

Use Case: Bill ChocAn for Services
Context: A provider carries out a service for a member and bills them the correct amount.
Actors: Provider
Main Success Scenario:
  1. After a service is provided to a member, the provider validates the member.
  2. The provider enters the date of the provided service to the terminal.
  3. The provider requests the Provider Directory and looks up the 6-digit code corresponding to the service.
  4. The provider keys the service code into the terminal.
    5. 4.1. The service corresponding to the service code is displayed on the screen.
    4.2. The provider is prompted to verify that the correct service is displayed.
  5. The provider optionally enters comments on the service.
  6. Information concerning the service provided is written to disk.
  7. The terminal displays the fee to be paid.
  8. The provider verifies the correct billing information and enters it a form.
Extensions:
    4a. The service code is invalid.
    4a.1. The software finds no match for the code given by the provider.
    4a.2. The terminal displays an error message.
    4a.3. The use case continues at step 3.

Glossary

Term Definition
Electronic Funds Transfer (EFT) Data Report A report that details the compensation to be paid to each provider's bank account.
Manager An individual who is able to receive a number of individual reports at their request and is in charge of managing accounts payable.
Member A person who is receiving services from ChocAn in exchange for a monthly fee.
Member Records A record containing the information off all members that can be updated by operators.
Member Report A report made for the member requesting ChocAn services that details information that pertains to them, such as their provider name and service name.
Operator A person who is able to modify the records of both providers and members.
Provider A healthcare professional who provides services to ChocAn members.
Provider Directory An alphabetically ordered list of service names and corresponding service codes and fees.
Provider Records A record containing the information of all providers that can be updated by operators.
Provider Report A report made for providers detailing the services they provided and their personal information, I.e. name, provider number, etc.
Summary Report A report detailing the accounts payable.
Timer A running clock system that activates the weekly accounting procedure.
